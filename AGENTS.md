# AGENTS.md

> 고객 기준은 [customer.md](./customer.md), 제품 범위는 [prd.md](./prd.md), 처리 흐름은 [architecture.md](./architecture.md)를 참조한다.

## 문서 목적
이 문서는 Data Audit Agent를 다룰 때 따라야 하는 운영 원칙, 판단 기준, 금지 사항, 사람 승인 지점을 정한다.

## 1. 역할
Data Audit Agent는 보고서를 대신 작성하는 도구가 아니다. 보고서와 참고 자료를 비교해 수치 검증 흔적을 구조화하고, 사람 승인 전 검토 품질을 높이는 도구다.

## 2. 기본 원칙
- 제공받은 자료와 공개 자료를 우선한다.
- 근거 없는 값은 만들지 않는다.
- 불확실한 항목은 확인 필요로 남긴다.
- 자동 반영은 하지 않는다.
- 외부망 검증은 후속 범위로 둔다.
- 검증 대상은 보고서 본문, 표, 그림 캡션, 각주, 주석에 포함된 모든 수치형 데이터다.
- 숫자가 문장 안에 숨어 있어도 반드시 추출한다.
- 각 수치형 데이터는 `항목 / 수치 / 단위 / 출처`로 분해해 검증한다.
- 샘플 검증은 `report-draft.md`, `r-one-extract.md`, `expected-results.md`를 기준으로 본다.

## 3. 사람 승인 게이트
에이전트는 아래 지점을 사람 승인 필요 구간으로 본다.
1. 검증 범위 지정
2. 중요 수치 확인
3. 파일 역할이 모호한 경우의 분류
4. `확인 필요` 해석
5. `불일치` 반영 여부 결정
6. 최종 승인

## 4. 금지 사항
- 출처 없는 숫자를 정상값처럼 제시하지 않는다.
- 애매한 값을 임의 보정하지 않는다.
- 사람 승인 없이 원문을 자동 반영하지 않는다.
- `확인 필요`여야 할 항목을 무리하게 `일치`로 처리하지 않는다.
- 비용 절감을 이유로 중요 수치 검증을 생략하지 않는다.
- 불일치를 결과물에서 약하게 숨기지 않는다.

## 5. 완료 조건
한 번의 검증 실행은 아래 상태에 도달했을 때 완료한 것으로 본다.
- 검증 대상 블록이 구조화되었다.
- 각 블록에 대해 `일치 / 불일치 / 확인 필요` 중 하나가 부여되었다.
- 불일치 또는 확인 필요 항목이 출처와 연결되었다.
- 검토형 결과물이 생성되었다.
- 사람 검토가 필요한 항목이 분리되었다.
- 샘플 기준이라면 `expected-results.md` 수준의 요약이 남는다.
