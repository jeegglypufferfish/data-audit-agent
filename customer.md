## 문서 목적
이 문서는 Data Audit Agent를 누가, 어떤 상황에서, 왜 쓰는지 정리한다.

## 1. 고객 정의
Data Audit Agent의 고객은 제출 직전 보고서 초안과 출처 자료를 검증하는 내부 실무자다.

## 2. MVP 기준 사용자
이번 MVP의 기준 사용자는 `보고서 작성자`다.

이유는 다음과 같다.

- 제출 직전 최종 점검에서 출처 재탐색과 수치 대조를 가장 많이 수행한다.
- 반복 검증의 부담을 가장 먼저 체감한다.
- 이 단계에서 검증 흔적이 구조화되면 이후 검토자와 최종 승인자의 재작업도 줄일 수 있다.

## 3. 핵심 사용자군
### 1차 사용자: 보고서 작성자
- 초안과 수치를 최종 점검하는 사람
- 출처를 다시 열지 않고 불일치를 빨리 찾고 싶다

### 2차 사용자: 검토자
- 승인 전 품질을 확인하는 사람
- 검증 흔적과 불일치 목록을 빠르게 확인하고 싶다

### 3차 사용자: 최종 승인자
- 반영 여부와 책임 경계를 확인하는 사람
- 현재 범위와 후속 범위를 분명히 보고 싶다

## 4. 사용 시점
이 도구는 `초안 작성 도중`보다 `제출 직전 최종 점검`에 쓰는 것이 맞다.

## 5. 핵심 JTBD
- 보고서, 참고 문헌, 내부 자료를 한 묶음으로 정리한다.
- 표, 본문, 그림, 그래프 숫자를 출처와 대조한다.
- 현재 범위와 후속 범위를 분리한다.
- 확인 필요 항목과 승인 흔적을 남긴다.

## 6. 주요 문제
- 출처가 여러 파일로 흩어져 재탐색이 반복된다.
- 검증 기준이 사람마다 달라 누락과 재작업이 생긴다.
- 무엇이 확인됐고 무엇이 남았는지 정리하기 어렵다.
- 승인 전 반영 범위가 흐려진다.

## 7. 기대 결과
- 검증 결과가 한눈에 정리된다.
- 출처와 근거 위치가 연결된다.
- 확인 필요와 불일치가 분리된다.
- 최종 승인 전에 사람이 판단할 수 있다.

## 8. 신뢰 조건
- 제공받은 자료와 공개 자료만 사용한다.
- 근거 없는 값은 만들지 않는다.
- 불확실한 항목은 확인 필요로 남긴다.
- 자동 반영은 하지 않는다.
- 외부망 검증은 후속 범위로 둔다.
